I first launched the robot with `ros2 launch limo_bringup limo_start.launch.py` and verified that the important topics were alive: `/scan`, `/odom`, `/tf`, `/tf_static`, `/imu`, and `/limo_status`. The `RobotModel` did not appear, but I ignored that because it was not important for mapping.

After that, I explored the installed `limo_bringup` package and discovered that the launch file I had first expected from the guide did not exist under the name `limo_cartographer.launch.py`; instead, the correct installed file was `cartographer.launch.py` inside `/home/agilex/limo_ros2_ws/install/limo_bringup/share/limo_bringup/launch`. I launched mapping with `ros2 launch limo_bringup cartographer.launch.py`, drove the robot around the room, and successfully built a map with Cartographer. I then saved the map into the package maps directory using `ros2 run nav2_map_server map_saver_cli -f ~/limo_ros2_ws/src/limo_ros2/limo_bringup/maps/fablab_map`, which created `fablab_map.pgm` and `fablab_map.yaml`.

Next, I launched navigation using my saved map with `ros2 launch limo_bringup limo_nav2.launch.py map:=/home/agilex/limo_ros2_ws/src/limo_ros2/limo_bringup/maps/fablab_map.yaml`. At first the map did not load because I had used `~` in the path, but using the full absolute path fixed that. Localization and navigation were still unstable, though: the laser did not align well with the saved map, the robot pose jumped around, and the path kept changing radically. I inspected the saved `.pgm` map and realized that black areas are the confident occupied walls, while gray areas are uncertain or unknown, which explains why the robot had trouble using the map reliably for localization and planning.

I also realized that the environment itself is part of the problem: there are **glass walls** in the fablab, and those can confuse a 2D lidar because the beams may pass through, reflect strangely, or return weakly. That means the map is not just a visualization for the human user; it is a real internal world model for the robot, and if important walls are missing or uncertain, localization and navigation suffer badly. Because of that, I decided that the next mapping attempts should focus on **smaller, cleaner areas with solid walls** instead of large messy zones with glass.

Later in the session I compared the stock AgileX Cartographer setup with a more tuned custom stack from another project. From that comparison, I identified useful improvements such as using `base_link` as the tracking frame, keeping odometry enabled, limiting lidar range more intelligently, and generally treating the vendor files as a baseline rather than the only truth. I created a new tuned Cartographer config file `~/limo_ros2_ws/src/limo_ros2/limo_bringup/config_files/limo_real_tuned.lua`, but when I launched Cartographer with that file it did not produce submaps, so I decided not to keep debugging that separate path for now. Instead, I went back to the stock file `limo_lds_2d.lua` and planned to modify it **one change at a time**, starting with changing `tracking_frame = "odom"` to `tracking_frame = "base_link"`.

So the main outcome of today is this: I stopped treating simulation as the main battlefield, moved back to the **real robot**, successfully built and saved a real map using the official AgileX workflow, launched navigation with my saved map, and then diagnosed why localization/navigation still felt unstable.
